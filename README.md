# Real-Time Multilingual Voice AI Agent
### Clinical Appointment Booking

A production-grade voice AI agent for healthcare appointment management, handling English, Hindi, and Tamil conversations in real time — with end-to-end response latency under 450ms.

---

## Architecture Overview

```
Twilio (WebSocket audio) → STT (Deepgram) → Agent (Claude) → TTS (ElevenLabs) → Twilio
                                                  ↕
                            Redis (session) + PostgreSQL/pgvector (long-term memory)
                                                  ↕
                              BullMQ job queue (outbound campaigns)
```

### Pipeline Stages & Latency Budget

| Stage | Target | Notes |
|---|---|---|
| Twilio → WebSocket | ~20ms | Local WebSocket handshake |
| STT first result (Deepgram streaming) | ~80ms | `nova-2` model, `endpointing: 300ms` |
| Language detection | ~5ms | Unicode script matching, no API call |
| LLM first token (Claude streaming) | ~150ms | `claude-sonnet-4-5`, max_tokens: 400 |
| TTS first audio chunk (ElevenLabs) | ~100ms | `eleven_turbo_v2`, sentence-boundary flush |
| WebSocket → Twilio | ~20ms | Audio chunk transmission |
| **Total** | **~375ms** | Under 450ms target |

Latency is measured per-call and logged via `LatencyTracker`. Every `mark()` call adds a timestamped entry to pino logs.

## Setup

### Prerequisites
- Node.js 20+
- PostgreSQL 15+ with pgvector extension
- Redis 7+
- Twilio account with a phone number
- Deepgram, ElevenLabs, Anthropic API keys

### Install

```bash
git clone <repo>
cd voice-agent
npm install

cp .env.example .env
# Fill in all API keys

# Enable pgvector in your PostgreSQL instance
psql -U postgres -c "CREATE EXTENSION IF NOT EXISTS vector;"

# Run migrations and seed
npm run db:migrate
npm run db:generate
npm run db:seed
```

### Local development with Twilio

```bash
# Start the server
npm run dev

# In another terminal, expose to internet (Twilio needs a public URL)
ngrok http 3000

# Set BASE_URL in .env to your ngrok URL
# Configure Twilio webhook: https://your-ngrok.ngrok.io/incoming-call
```

## Memory Design

### Two-tier memory architecture

**Session memory (Redis, TTL 2h)**
- In-call state: current language, pending confirmations, intent tracking
- Conversation history for the active call
- Survives brief disconnections within the session window
- Key: `session:{callSid}`, auto-expires

**Long-term memory (PostgreSQL + pgvector)**
- Patient preferences (language, preferred doctor, time preferences)
- Interaction summaries (auto-generated by Claude Haiku after each call)
- Appointment history
- Semantic search: query embeddings stored as `vector(1536)`, searched via cosine similarity (`<=>` operator)

**Retrieval flow:**
1. On call start → load patient record + top-5 recent memories by recency
2. During complex queries → semantic search over past interactions (pgvector)
3. Both injected into system prompt as structured context
4. On call end → Claude Haiku generates a 1-2 sentence summary → stored with embedding

## Multilingual Support

Language is detected per utterance using Unicode script ranges:
- Devanagari (U+0900–U+097F) → Hindi
- Tamil (U+0B80–U+0BFF) → Tamil
- Fallback → English

Deepgram's `language: 'multi'` model handles transcription across all three. TTS uses separate ElevenLabs voices per language. System prompt switches language on detection, and preference is persisted to the patient record for future calls.

## Scheduling & Conflict Logic

All appointment operations are implemented as Claude tool calls (never hardcoded):

- `check_available_slots` — queries DB with doctor/specialty/date filters
- `book_appointment` — transactional: checks slot availability + creates appointment atomically
- `cancel_appointment` — marks appointment cancelled, frees slot
- `reschedule_appointment` — swaps slots in a single transaction
- `find_alternative_slots` — finds next N available slots when requested slot is gone
- `list_doctors` — filtered doctor lookup
- `list_patient_appointments` — patient's upcoming bookings

**Conflict prevention:**
- Double-booking: slot marked `isBooked` in DB; `book_appointment` re-checks inside a transaction
- Past slot selection: rejected at tool level with a clear error; agent offers alternatives
- Unavailable doctor: `find_alternative_slots` called automatically

## Barge-in / Interrupt Handling

When VAD detects the patient speaking mid-response:
1. `AbortController.abort()` stops the LLM stream mid-generation
2. `sendClearAudio()` sends Twilio's `clear` event to flush buffered audio
3. STT transcript of the new utterance starts a fresh agent turn

## Outbound Campaigns

```
POST /outbound/reminder/:appointmentId   — schedule a reminder call
POST /outbound/follow-up                 — schedule follow-ups for recent appointments
GET  /outbound/status/:jobId             — check job status
```

Jobs are queued in BullMQ (Redis-backed), with:
- Configurable delay (default: 24h before appointment)
- 3 retry attempts with exponential backoff
- Twilio AMD (Answering Machine Detection) to skip voicemail
- Outcome logged back to `CampaignLog` table

The outbound call uses the same voice pipeline as inbound — the agent adapts its opening based on `campaignType` passed as a URL parameter.

## Tradeoffs & Known Limitations

| Tradeoff | Decision | Reason |
|---|---|---|
| Sentence-boundary TTS flush | Accumulate text until `.!?` then flush | Balances latency vs naturalness; sending word-by-word causes choppy audio |
| Deepgram `multi` model | Single model for all languages | Avoids per-utterance language detection round trip; slightly lower accuracy vs language-specific model |
| Claude Haiku for summaries | Haiku not Sonnet | 10x cheaper, fast enough for async post-call task |
| pgvector vs dedicated vector DB | pgvector | Keeps infra simple; sufficient for clinic-scale data |
| 400 token max response | Short LLM responses | Direct latency impact; voice responses should be concise anyway |

### Known Limitations
- ElevenLabs Hindi/Tamil voice quality depends on available multilingual voice IDs; may need custom voice cloning
- pgvector semantic search requires OpenAI embeddings (additional API dependency); can be swapped for any 1536-dim embedding model
- Twilio AMD adds ~3-5s delay on outbound calls; cannot be parallelized
- No WebRTC VAD — relies on Deepgram's endpointing; barge-in detection latency ~300ms

## Evaluation Metrics

Every call logs:
- `stt_final` → `llm_start` gap
- `llm_start` → `tts_start` gap (first token)
- `tts_start` → `tts_done` gap
- Total end-to-end latency

Search logs with `grep "Pipeline latency"` or ship to Datadog/Grafana via pino transport.
